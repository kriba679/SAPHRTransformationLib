# 04.1 SAP Cloud Transport Management (cTMS) — Full Configuration Guide

**Scenario:** Two BTP Trial Global Accounts. Global Account A (Source) has **two subaccounts** — a dedicated Central Services subaccount for cTMS, and a DEV subaccount for Integration Suite. Global Account B (Target) has one PROD subaccount with Integration Suite. This reflects the SAP best practice architecture.

---

## Architecture — Best Practice (3 Subaccounts across 2 Global Accounts)

```
Global Account A (Source)                    Global Account B (Target)
│                                            │
├── Subaccount: Central Services             └── Subaccount: PROD
│   ├── Cloud Transport Management               ├── Integration Suite (CPI)
│   │   (subscription + service instance)        ├── Content Agent Service
│   └── Destination: CAS_on_target               └── Process Integration Runtime
│       (points to CAS in PROD subaccount)
│
└── Subaccount: DEV
    ├── Integration Suite (CPI)
    ├── Content Agent Service
    ├── Process Integration Runtime
    └── Destinations:
        ├── ContentAssemblyService → DEV CAS
        ├── CloudIntegration → DEV PIRuntime
        └── TransportManagementService → cTMS in Central Services
```

> **Why Central Services is a dedicated subaccount:** cTMS manages the entire transport landscape (DEV node, PROD node, routes). It should not "belong" to the DEV environment. Separating it means: (1) the DEV subaccount can be torn down and rebuilt without destroying the transport infrastructure; (2) transport operators get access to cTMS independently from who has DEV subaccount access; (3) cTMS can serve multiple application subaccounts in future without being tied to any one of them.

> **Cross-Global-Account:** The steps are technically identical to the official SAP tutorial (which uses two subaccounts within the same global account). OAuth2 API calls are HTTPS — they cross global account boundaries without any special configuration. Constraints: (1) all subaccounts must be in the **same CF region** (e.g., all `us10`); (2) an admin of Global Account B must perform the PROD-side steps.

## End-to-End Flow

```
Developer clicks Transport on a CPI package (DEV subaccount)
  → CPI calls CAS (DEV) via ContentAssemblyService destination
  → CAS (DEV) assembles the package into an MTAR archive
  → CAS (DEV) calls cTMS API via TransportManagementService destination
  → cTMS (Central Services) places TR in DEV_Node queue → auto-forwards to PROD_Node queue
  → Transport Admin opens cTMS UI (Central Services), imports from PROD_Node queue
  → cTMS calls CAS (PROD) via CAS_on_target destination (OAuth2, cross-global-account)
  → CAS (PROD) deploys the MTAR into PROD CPI via its local CloudIntegration destination
  → Package appears in PROD CPI tenant (undeployed — iFlows must be deployed manually)
```

## Authentication & Authorization — Full Explanation

### OAuth 2.0 Client Credentials Flow (Machine-to-Machine)

Every service-to-service call in this setup uses **OAuth 2.0 Client Credentials Grant**. There is no human login involved:

1. The calling service (e.g., CAS) sends its **Client ID + Client Secret** to the **XSUAA token endpoint** (`/oauth/token`).
2. XSUAA validates the credentials and returns a short-lived **JWT Bearer Token**.
3. The calling service includes this JWT in `Authorization: Bearer <token>` when calling the target API.
4. The target service validates the JWT (issuer, expiry, scopes) before processing the request.

BTP Destinations of type `OAuth2ClientCredentials` automate steps 1–3. You configure the credentials once; BTP handles the token exchange at runtime.

### What a Service Key Contains

When you create a **Service Instance**, BTP provisions that service and registers an OAuth client with XSUAA. When you create a **Service Key**, XSUAA generates a Client ID + Secret bound to that instance:

```json
{
  "uri": "https://<service-api-endpoint>",      ← use this for the Destination URL
  "url": "https://<service-api-endpoint>",      ← some services use "url" instead of "uri"
  "uaa": {
    "clientid":     "sb-<uuid>!b<number>",
    "clientsecret": "<secret>",
    "url":          "https://<subdomain>.authentication.<region>.hana.ondemand.com",
    "tokenurl":     "https://<subdomain>.authentication.<region>.hana.ondemand.com/oauth/token"
  }
}
```

> ⚠️ **cTMS service key specifically uses `uri`** (not `url`) for the service API endpoint. Scroll to the **bottom** of the cTMS service key JSON to find `uri`. Using the top-level `url` points to XSUAA, not cTMS.

### XSUAA Role Collections for cTMS

Two role collections are **pre-delivered** (exist after subscription, just assign users). Two must be **created manually** and populated via the subscription Roles tab.

| Role Collection | Pre-delivered? | Who gets it | What it allows |
|---|---|---|---|
| `TMS_LandscapeOperator_RC` | ✅ Yes | Basis/Admin | Create/edit nodes, routes, destinations |
| `TMS_Viewer_RC` | ✅ Yes | Audit/read-only | View landscape and import queues |
| `TMS_Administrator_RC` | ❌ Create manually | Overall admin | All permissions combined |
| `TMS_TransportOperator_RC` | ❌ Create manually | Dev/Ops team | Trigger imports; can be restricted to specific nodes via attributes |

The `TransportOperator` role template supports **node-level attributes** — you can restrict an operator to only `DEV_Node` (developers) or only `PROD_Node` (ops team), enforcing separation of duties. See Step 5c for details.

CPI-side roles (assigned in DEV subaccount):
- `AuthGroup.Administrator` → unlocks `Settings → Integrations → Transport` in CPI
- `WorkspacePackagesTransport` → shows the Transport button on packages in CPI Design

### Why 401 on Connection Test is Normal

BTP destination connection tests only check network reachability — they do not perform an OAuth token exchange. The request arrives at the service without a valid Bearer token, correctly returning `401`. Ignore 401 on all connection tests in this guide. Validate instead via **Check Configuration** in the CPI Settings page.

### Why You Must Log Out After Role Assignment

XSUAA JWTs are stateless and self-contained — new role scopes are only included in a token generated at your next login. After assigning any role collection, log out and log back in before testing.

### Step 2: Assign Entitlements

Navigate to: **Global Account A → Entitlements → Entity Assignments**

| Service | Plan | Type | Purpose |
|---|---|---|---|
| Integration Suite | trial | Application | CPI tenant (already done) |
| Content Agent Service | standard | Service | Assembles and exports MTAR |
| Process Integration Runtime | api | Service | CPI transport API access |

| Service | Plan | Type | Purpose |
|---|---|---|---|
| Integration Suite | trial | Application | CPI tenant |
| Content Agent Service | standard | Service | Deploys MTAR into PROD CPI |
| Process Integration Runtime | api | Service | CPI deployment API access |

**Why two plans for cTMS?**
The **Application** plan provides the Fiori UI for landscape management and import queue operations. The **Standard** plan provides the API endpoint (`uri` in the service key) that CAS calls to push transport requests programmatically. Both are required — the UI for humans, the service instance for machine-to-machine communication.

### Step 8: Create PIRuntime Instance + Service Key (DEV)

Navigate to: **DEV Subaccount → Services → Service Marketplace → Process Integration Runtime**

1. Plan: **api** | Instance name: e.g., `pi-dev`
2. In the JSON configuration, provide **all three roles**:

```json
{
  "roles": [
    "WorkspacePackagesTransport",
    "WorkspacePackagesRead",
    "WorkspacePackagesEdit"
  ]
}
```

3. Create service key: e.g., `pi-on-cpi-dev`

Note for Step 10:
```
url              → CPI tenant base URL
uaa.clientid     → OAuth Client ID
uaa.clientsecret → OAuth Client Secret
uaa.tokenurl     → Full token URL (already includes /oauth/token for PIRuntime keys)
```

**Why three roles?**
- `WorkspacePackagesTransport` — permits the transport API call
- `WorkspacePackagesRead` — permits CAS to read package content for MTAR assembly
- `WorkspacePackagesEdit` — permits CAS to update transport metadata during export

All three scopes are embedded in every JWT token issued to this OAuth client. Missing any one causes a `403 Forbidden` on the specific API call that requires it. If the instance was created with fewer roles, you must **recreate it** (JSON config cannot be updated after creation).

### Step 9: Create `ContentAssemblyService` Destination (DEV)

Navigate to: **DEV Subaccount → Connectivity → Destinations → Create**

Choose **Service Instance** method (not "From Scratch") — this auto-populates all OAuth fields from the service key, eliminating manual copy errors.

- Select instance: `cpi-dev` (your DEV CAS instance)
- Select service key: `cas-on-cpi-dev`
- Click **Next → Next**

In the Destination Details step:

| Field            | Value                                                        |
| ---------------- | ------------------------------------------------------------ |
| **Name**         | `ContentAssemblyService` (**exact case — hardcoded in CPI**) |
| Description      | Destination to SAP Content Agent (DEV)                       |
| All other fields | Auto-populated from service key — leave unchanged            |

Click **Create**.

**![[Pasted image 20260618114359.png]]
**

**Why this exact name?** CPI's transport module looks up a BTP destination named exactly `ContentAssemblyService` in its own subaccount at runtime. A single character difference (including casing) causes CPI to fail with "destination not found" when the developer clicks Transport.

| Client Secret     | `uaa.clientsecret` from DEV PIRuntime service key                           |
| Token Service URL | `uaa.tokenurl` from DEV PIRuntime service key                               |

Click **Create**.
![[Pasted image 20260618115032.png]]


**Why append `/api/1.0/transportmodule/Transport`?** The service key `url` gives the CPI tenant root. CAS calls this specific API path on CPI to trigger package assembly into MTAR format. There is no way to infer this path from the service key — it must be constructed manually.

## Global Account A — Part 3: Connect Landscape in Central Services

### Step 15: Create `CAS_on_target` Deployment Destination (Central Services)

Navigate to: **Central Services Subaccount → Connectivity → Destinations → Create → From Scratch**

Use the **PROD CAS service key values** from Step 12:

| Field | Value |
|---|---|
| Name | `CAS_on_target` (not a fixed name — you reference it when configuring the PROD node in Step 16) |
| Description | Content Agent on PROD (Global Account B) |
| URL | `url` from PROD CAS service key |
| Proxy Type | Internet |
| Authentication | `OAuth2ClientCredentials` |
| Client ID | `uaa.clientid` from PROD CAS service key |
| Client Secret | `uaa.clientsecret` from PROD CAS service key |
| Token Service URL | `uaa.url` from PROD CAS service key + `/oauth/token` |

Click **Create**.

![[Pasted image 20260618120207.png]]


**What does this destination do?** When the transport admin imports a TR from the PROD_Node queue, cTMS calls this destination to deliver the MTAR to CAS in Global Account B. This OAuth2 call crosses the global account boundary — which works because it is simply an HTTPS API call to the CAS service URL in Global Account B. No special cross-account configuration is needed.

**Why is this destination in Central Services (not DEV)?** Because cTMS lives in Central Services. The deployment destination is used by cTMS at import time to call the target endpoint — so it must exist in the same subaccount as cTMS.

### Step 16: Configure Transport Nodes and Route (cTMS UI)

Open the **cTMS UI**: Central Services → Services → Instances and Subscriptions → Cloud Transport Management → Go to Application.

Navigate to **Landscape Visualization**.

**Create the Source Node (DEV_Node):**

Click **+** (Create a Node):

| Field | Value | Notes |
|---|---|---|
| Name | `DEV_Node` | Must exactly match the `sourceSystemId` value you set in Step 17 |
| Description | SAP Integration Suite — DEV (Global Account A) | |
| **Allow Upload to Node** | **Checked** | Permits CAS (source) to push transport requests into this node's queue. Without this, the upload API call is rejected. |
| Content Type | Leave blank | Source nodes do not deploy — they only receive uploads and forward |
| Destination | Leave blank | Only target nodes need a deployment destination |

Click **OK**.

**Create the Target Node (PROD_Node):**

Click **+** (Create a Node):

| Field | Value | Notes |
|---|---|---|
| Name | `PROD_Node` | |
| Description | SAP Integration Suite — PROD (Global Account B) | |
| Allow Upload to Node | Leave unchecked | Pure target node |
| **Content Type** | **Multi-Target Application** | Integration Suite packages are transported as MTAR archives |
| **Destination** | **`CAS_on_target`** | Created in Step 15 — points to CAS on the PROD subaccount |

Click **OK**.

**Why Content Type = Multi-Target Application?** This tells cTMS the format of the archives it will receive and how to route them for deployment. MTAR is the archive format used by CF-based services including Integration Suite.

**Create the Transport Route:**

Click the connector icon → Create Route:

| Field | Value |
|---|---|
| Name | `DEV_to_PROD` |
| Description | Route from DEV to PROD |
| Source Node | `DEV_Node` |
| Target Node | `PROD_Node` |

Click **OK**.

![[Pasted image 20260618120940.png]]


**What does the route do?** When CAS uploads an MTAR to `DEV_Node`'s queue, cTMS automatically forwards it to `PROD_Node`'s import queue via this route. The import itself is triggered manually by the transport admin — the route defines the direction, not the timing.

### Step 17: Create `TransportManagementService` Destination (DEV Subaccount)

> Note: this destination is created in the **DEV subaccount** (where CAS lives), using credentials from the **cTMS service key** in Central Services.

Navigate to: **DEV Subaccount → Connectivity → Destinations → Create → From Scratch**

| Field | Value |
|---|---|
| **Name** | `TransportManagementService` (**exact case — hardcoded in CAS**) |
| Description | Destination to SAP Cloud Transport Management |
| URL | **`uri`** from cTMS service key (Central Services, Step 4) — scroll to **bottom** of key JSON |
| Proxy Type | Internet |
| Authentication | `OAuth2ClientCredentials` |
| Client ID | `uaa.clientid` from cTMS service key |
| Client Secret | `uaa.clientsecret` from cTMS service key |
| Token Service URL | `uaa.url` from cTMS service key + `/oauth/token` |
| **Additional Property** | Key: `sourceSystemId` / Value: `DEV_Node` |

Click **Create**.

![[Pasted image 20260618121628.png]]

**What is `sourceSystemId`?** When CAS calls the TMS API to upload a transport request, TMS must know which node the upload belongs to so it can route it downstream via the configured route. This value must exactly match the Name field of the source transport node in cTMS (`DEV_Node` from Step 16). A mismatch causes TMS to reject the upload with an unrecognized source system error.

**Why is this destination in DEV, even though cTMS is in Central Services?** CAS (in DEV) is the service making the API call to cTMS. BTP Destination service looks up destinations in the subaccount of the calling application. So the `TransportManagementService` destination must be in the same subaccount as CAS — the DEV subaccount. The destination points to a service (cTMS) that lives in a different subaccount, which is perfectly valid.

### Step 19: Enable Transport Mode in CPI (DEV Subaccount)

Open the **DEV CPI tenant** via Integration Suite.

Navigate to: **Settings → Integrations → Transport tab**

1. Click **Edit** (bottom-right).
2. Change dropdown from (no selection) to **Transport Management Service**.
3. Click **Save** (bottom-right).

Click **Check Configuration**. Expected result:
```
✓ ContentAssemblyService      → SUCCESS
✓ CloudIntegration            → SUCCESS
✓ TransportManagementService  → SUCCESS (DEV_Node reachable in Central Services cTMS)
```

![[Pasted image 20260618130502.png]]

If any check fails — error messages identify the specific destination issue. Common causes: wrong destination name casing, wrong field from service key (`url` vs `uri` for TMS), PIRuntime instance missing required roles.

### Step 20: Run a Test Transport

**In the DEV CPI tenant:**

1. Go to **Discover → Integrations** → find any standard SAP package → click **Copy** to add it to your Design section.
2. Go to **Design → Integrations and APIs** → open the copied package.
3. Click **Transport** (top-right — only visible if `WorkspacePackagesTransport` role is assigned).
4. Enter a comment (e.g., *Test transport 01*), optionally check *Propagate logged-in user as transport owner*.
5. Click **Transport**. A Transport Request (TR) number is returned.

**In the cTMS UI (Central Services subaccount):**

1. Home screen shows a pending count on `PROD_Node`. Click it to open the import queue.
2. Select the Transport Request. Verify Transport Description (package name) and Owner (your email).
3. Click **Import All** → confirm with **OK**.
4. Status changes to `Succeeded`. Click the log icon to see detailed import steps.

**In the PROD CPI tenant:**

Go to **Design → Integrations and APIs**. The transported package appears. The creator shown will be the Client ID of the PROD CAS service key — confirming automated OAuth2 deployment by CAS (not a manual import). Artifacts arrive **undeployed** — deploy iFlows manually in the PROD tenant.

## Appendix B — Common Errors and Root Causes

| Error | Root Cause | Fix |
|---|---|---|
| `ContentAssemblyService not found` | Destination name casing wrong | Must be exactly `ContentAssemblyService` |
| `TransportManagementService fails` | Used `url` instead of `uri` from cTMS key | Edit destination; use `uri` at bottom of cTMS service key JSON |
| `sourceSystemId not recognized` | Additional property doesn't match node name | Must be exactly `DEV_Node` (matches Step 16 node name) |
| `Configuration check fails for CloudIntegration` | PIRuntime instance missing roles | Recreate instance with all 3 roles in JSON config |
| `Import fails — deployment error` | `CloudIntegration` missing in PROD subaccount | Create `CloudIntegration` destination in PROD using PROD PIRuntime key |
| `Import fails — CAS auth error` | `CAS_on_target` has wrong key values | Verify PROD CAS service key is used, not DEV CAS key |
| `Transport button not visible in CPI` | `WorkspacePackagesTransport` role not assigned | Create role collection, assign role + user, log out and back in |
| `Check Configuration does not show Transport tab` | Settings path wrong | Path: `Settings → Integrations → Transport tab` (not `Settings → Transport`) |
| `FileStorage / region error on import` | Subaccounts in different CF regions | All three subaccounts must be in the same CF region (e.g., all `us10`) |
| `401 on connection test` | Expected — test doesn't send a Bearer token | Ignore; validate via Check Configuration in CPI |
